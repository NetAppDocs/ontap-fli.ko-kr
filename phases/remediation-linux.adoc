= ONTAP FLI Linux 호스트 문제 해결
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Linux 호스트에서 ONTAP FLI 마이그레이션을 실행하기 전에 다음 사전 마이그레이션 단계를 완료하십시오. 오프라인 마이그레이션과 온라인 마이그레이션에 대한 단계가 각각 다릅니다.



== 사전 마이그레이션 - 오프라인 FLI

이러한 단계는 온라인 및 오프라인 FLI 모두에 공통적으로 적용됩니다. 호스트에서 소스 LUN의 프로비저닝을 해제하기 전에 다음 단계를 완료했는지 확인하십시오.

[cols="3"]
|===
| 파일 시스템(LVM 및 Non-LVM) | LVM이 있는 Pacemaker(SCSI3 PR) | SANBOOT 


| 1단계: 엔드 투 엔드 블록 장치 구성 수집: 다중 경로 상태/경로, 블록 장치의 LVM 레이아웃 및 올바른 부팅 장치와 부팅 매개변수를 확인하는 GRUB/부트로더 설정. | 1단계: 엔드 투 엔드 블록 장치 구성 수집: 다중 경로 상태/경로, 블록 장치의 LVM 레이아웃 및 올바른 부팅 장치와 부팅 매개변수를 확인하는 GRUB/부트로더 설정. | 1단계: 엔드 투 엔드 SAN 부팅 구성: 다중 경로 상태/경로, 부팅 LUN의 LVM 레이아웃, 그리고 올바른 부팅 장치와 부팅 매개변수를 확인하는 GRUB/부트로더 설정을 수집합니다. 


| 2단계: 각 데이터 장치(LVM/비LVM)에 대한  `/etc/fstab` 항목이 해당 FS UUID로 참조되는지 확인하십시오. | 2단계: 마이그레이션할 각 LUN에 대한 클러스터 리소스 그룹을 비활성화합니다. 비활성화 후 리소스 그룹을 삭제합니다. | 2단계: 데이터 장치(lvm/nonlvm)를 포함한 SAN 부팅 장치에 대한  `/etc/fstab` 항목이 해당 FS UUID로 참조되는지 확인하십시오. 


| 3단계: 대상 마운트 지점에 대한 모든 작업이 일시 중지되었는지 확인합니다. 블록 장치에 액세스하는 프로세스가 남아 있지 않은지 확인합니다. 필요한 경우 열려 있는 프로세스 핸들을 비우거나 종료합니다. | 3단계: 방금 비활성화/삭제한 리소스 그룹에 대한 LVM VG 구성 백업을 파일로 수행합니다. 그런 다음 VG를 비활성화합니다. | 3단계:  `GRUB_CMDLINE_LINUX` 줄을 올바른 UUID 참조로 업데이트했는지 확인하십시오. UUID 장치 참조를 기반으로 새 GRUB 설정을 다시 생성하십시오. 


| 4단계: 파일 시스템을 마운트 해제하고 LVM 볼륨 그룹 구성 백업을 수행합니다. VG를 비활성화하고 VG를 내보냅니다 | 4단계: PCS stonith 구성에서 소스 LUN을 제거합니다. | 4단계: 새 grub 설정으로 변경 사항을 검증하고 SAN 부팅 장치에 대한 변경 사항을 반영하여 새 initrd 이미지를 다시 생성합니다. 


| 5단계: 장치의 다중 경로 구성을 플러시하고 오래된 SCSI 장치 파일을 모두 제거합니다. | 5단계: 장치의 다중 경로 구성을 플러시하고 오래된 SCSI 장치 파일을 모두 제거합니다. 참고: LUN 영구 예약이 발견되면 이를 지우고 PR 키가 남아 있지 않은지 확인하십시오. | 5단계: 호스트를 재부팅하여 위의 변경 사항이 정상적으로 적용되는지 확인합니다. 호스트가 정상적으로 부팅되면 FLI 마이그레이션을 위한 호스트 사전 복구 작업이 완료된 것입니다. 


| 6단계: 호스트에 업데이트된 장치 목록이 표시되도록 다중 경로 구성을 다시 로드합니다. 필요한 경우 호스트 SCSI 버스 재검색을 수행합니다. | 6단계: 호스트를 다시 스캔하여 기존 디바이스 매핑에 오래된 항목이 없는지 확인합니다. | 6단계: 이제 ONTAP FLI 워크플로우를 진행하기 위해 sanboot 호스트를 종료합니다. 
|===


== 사전 마이그레이션 - 온라인 FLI

아래는 온라인 FLI의 구체적인 단계이지만, 이러한 단계는 마이그레이션 전 오프라인 FLI의 단계를 먼저 완료해야 합니다. 다음 단계는 NetApp LUN이 호스트에 프로비저닝된 후에 수행해야 합니다. 이 단계는 NetApp LUN이 호스트에 프로비저닝된 후에 수행해야 합니다. 그리고 온라인 FLI에만 적용됩니다.

[cols="3"]
|===
| 파일 시스템(LVM 및 Non-LVM) | LVM이 있는 Pacemaker(SCSI3 PR) | SANBOOT 


| 1단계: 매핑된 모든 Linux 호스트에서 스토리지를 다시 검색합니다. | 1단계: 매핑된 모든 Linux 호스트에서 스토리지를 다시 검색합니다. | 1단계: 서버의 전원을 켭니다. 


| 2단계: 다중 경로 디바이스를 검색합니다. | 2단계: 다중 경로 장치를 검색합니다. 마이그레이션된 LUN이 PCS Stonith 구성에 추가되었는지 확인합니다. | 2단계: 서버 HBA BIOS 또는 UEFI에서 이 호스트 HBA에 제공되는 새 부팅 장치를 구성합니다. 


| 3단계: 볼륨 그룹을 가져오고 VG 활성화를 수행합니다. | 3단계: 새로 발견된 VG/LVM에 대해 VG 구성 백업 복원을 수행한 후 VG를 활성화합니다. | 3단계: LUN 가져오기의 일부인 새로 매핑된 LUN에서 부팅하도록 호스트를 다시 시작합니다. 


| 4단계: LVM 구성을 해당 마운트 지점에 마운트합니다. | 4단계: 새 PCS 리소스 그룹을 생성하고 사용 가능한 LV 및 파일 시스템에 해당하는 리소스를 추가합니다. 참고: 새로 활성화된 LVM 객체는 LV 및 파일 시스템 구조를 그대로 유지합니다. |  


|  | 5단계: 모든 리소스에 대한 PCS 클러스터 상태가 온라인이고 사용 가능한 상태인지 확인합니다. |  
|===
